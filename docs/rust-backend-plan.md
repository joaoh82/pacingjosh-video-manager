# Rust Backend Rewrite Plan

## Goal

Rewrite the Python/FastAPI backend as a Rust application in `backend-rust/` that is API-compatible with the existing Python backend. Every endpoint, request/response shape, and behavior must match so the existing Next.js frontend works without changes.

---

## Technology Choices

| Concern | Crate | Why |
|---|---|---|
| HTTP framework | **Actix-web 4** | Mature, high-performance, good Tauri compatibility, sync-friendly |
| Database ORM | **Diesel** (SQLite) | Compile-time query checking, strong SQLite support, migration tooling |
| Serialization | **serde / serde_json** | Standard Rust serialization |
| Request validation | **actix-web extractors + serde** | Deserialize + validate in one step |
| Async runtime | **Tokio** (via actix-web) | Required by actix-web |
| FFmpeg/ffprobe | **std::process::Command** | Direct subprocess calls, same as Python version |
| UUID | **uuid** | Scan ID generation |
| Date/time | **chrono** | DateTime handling, ISO 8601 parsing |
| File serving | **actix-files + manual streaming** | Range requests, thumbnail serving |
| CORS | **actix-cors** | Middleware for CORS headers |
| Config | **serde_json + std::env** | JSON config persistence + env vars, matching Python behavior |
| Logging | **env_logger + log** | Simple structured logging |

## Implementation Phases

### Phase 1: Project Scaffold & Database

**What**: Set up the Cargo project, Diesel, models, and migrations.

**Steps**:
1. `cargo init backend-rust`
2. Add all dependencies to `Cargo.toml`
3. Configure `diesel.toml` for SQLite
4. Write the Diesel migration (`up.sql`) creating all 6 tables with exact same columns, types, indexes, constraints, and foreign keys as Python
5. Implement `schema.rs` (auto-generated by `diesel print-schema`)
6. Implement `db.rs` — connection pool (`r2d2`), `init_db()` that runs migrations
7. Implement `models/` — Queryable, Insertable, AsChangeset structs for all tables
8. Implement `config.rs` — Settings struct, ConfigManager with JSON persistence + env fallback

**Tables to create** (matching Python exactly):
- `videos` (id, file_path UNIQUE, filename, duration, file_size, resolution, fps, codec, created_date, indexed_date, thumbnail_count)
- `metadata` (id, video_id UNIQUE FK→videos CASCADE, category, location, notes)
- `tags` (id, name UNIQUE)
- `video_tags` (video_id FK→videos CASCADE, tag_id FK→tags CASCADE, PK(video_id, tag_id))
- `productions` (id, title UNIQUE, platform, link, is_published)
- `video_productions` (video_id FK→videos CASCADE, production_id FK→productions CASCADE, PK(video_id, production_id))

**Indexes**: Same named indexes as Python version.

**Validation**: Run `diesel migration run` and verify schema matches.

### Phase 2: Core Services

**What**: Implement business logic services that mirror the Python service layer.

**Steps**:
1. `utils.rs` — `is_video_file()`, `get_file_size()`, `get_file_creation_date()`, `validate_directory()`, `is_path_safe()`, `format_file_size()`, `format_duration()`, `get_thumbnail_path()`, `ensure_directory_exists()`
2. `ffmpeg_service.rs`:
   - `check_ffmpeg()` — verify ffmpeg/ffprobe in PATH
   - `extract_metadata(path)` — run `ffprobe -v quiet -print_format json -show_format -show_streams`, parse JSON, extract duration/file_size/codec/resolution/fps/created_date
   - `generate_thumbnails(path, video_id, thumb_dir, count, width)` — run `ffmpeg -ss {ts} -i {path} -vframes 1 -vf scale={w}:-1 -y {out}` for each, 30s timeout
   - FPS parsing: handle "30000/1001" fractions and plain decimals
3. `video_service.rs` — all CRUD operations matching Python:
   - `get_video()`, `get_video_by_path()`, `update_video()`, `delete_video()`
   - `bulk_update_videos()` — field updates, add/remove tags, add/remove productions
   - Tag helpers: `get_or_create_tag()`, `update_video_tags()`, `add_tags_to_video()`, `remove_tags_from_video()`
   - Production helpers: `update_video_productions()`, `add_productions_to_video()`, `remove_productions_from_video()`
4. `search_service.rs`:
   - `search_videos()` — LIKE filters on filename/location/notes, category exact match, AND-logic tag filtering, production filtering, date range, sorting with null handling, pagination
   - `get_recent_videos()` — order by indexed_date DESC
   - `get_statistics()` — COUNT, SUM aggregations
   - `get_all_tags()` — LEFT JOIN + GROUP BY + COUNT
   - `get_all_categories()` — GROUP BY category + COUNT
5. `production_service.rs` — CRUD matching Python exactly
6. `scanner.rs`:
   - `ScanProgress` struct with same fields (scan_id, status, total, processed, successful, failed, skipped, current_file, errors, start_time, end_time)
   - `to_dict()` with dynamic elapsed_seconds and eta_seconds
   - `start_scan()` — validate directory, generate UUID, spawn `std::thread::spawn` (or `tokio::task::spawn_blocking`), return scan_id
   - `_run_scan()` / `_scan_directory()` — recursive file walk with `walkdir` crate, skip already-indexed, process each file, commit at end
   - `_process_video()` — skip files < 1KB, extract metadata, create Video + Metadata records
   - Thread-safe progress: `Arc<Mutex<HashMap<String, ScanProgress>>>`

**Validation**: Unit tests for ffprobe JSON parsing, FPS parsing, path safety, file utils.

### Phase 3: API Routes

**What**: Wire up all HTTP endpoints matching the Python API exactly.

**Steps**:
1. `main.rs`:
   - Actix-web server on configurable host:port (default 127.0.0.1:8000)
   - CORS middleware with same origins
   - App state: DB pool + scan progress map + config
   - Root (`/`) and health (`/health`) endpoints
   - Mount all route modules under `/api`

2. `config_routes.rs`:
   - `GET /api/config` — return configured status, video_directory, formats, thumbnail_count
   - `POST /api/config` — save config, return status
   - `OPTIONS /api/config` — CORS preflight
   - `GET /api/browse-folder` — platform-specific folder picker (Windows: PowerShell `System.Windows.Forms.FolderBrowserDialog`, macOS: osascript, Linux: zenity)

3. `videos.rs`:
   - `GET /api/videos` — search/filter/sort/paginate, response shape: `{ videos, total, page, limit, pages }`
   - `GET /api/videos/{id}` — single video with relationships
   - `PUT /api/videos/{id}` — update metadata/tags/productions
   - `DELETE /api/videos/{id}` — cascade delete
   - `POST /api/videos/bulk-update` — bulk operations
   - `GET /api/videos/recent/list` — recent by indexed_date
   - `GET /api/videos/stats/summary` — aggregate statistics
   - `POST /api/videos/{id}/open-folder` — open native file explorer

4. `tags.rs`:
   - `GET /api/tags` — tags with video counts
   - `GET /api/tags/categories` — categories with counts

5. `scan.rs`:
   - `OPTIONS /api/scan` — CORS preflight
   - `POST /api/scan` — start scan with directory
   - `GET /api/scan/status/{scan_id}` — progress with ETA
   - `POST /api/scan/rescan` — rescan configured directory

6. `stream.rs`:
   - `GET /api/thumbnails/{video_id}/{index}` — serve JPEG, Cache-Control 24h
   - `GET /api/stream/{video_id}` — video streaming with Range header support (206 Partial Content), 8KB chunks, proper content-type mapping

7. `productions.rs`:
   - `GET /api/productions` — list with video counts
   - `POST /api/productions` — create (201)
   - `PUT /api/productions/{id}` — update
   - `DELETE /api/productions/{id}` — delete with cascade

**Response shape rules** (must match Python exactly):
- Video responses include nested `metadata` (aliased from `video_metadata`), `tags` array, `productions` array
- Tag responses include `count` field
- Production responses include `video_count` field
- Scan status includes computed `elapsed_seconds` and `eta_seconds`
- Error responses use `{ detail: string }` format (FastAPI convention)

**Validation**: Test every endpoint with the existing Next.js frontend.

## Key Implementation Notes

### Database Differences
- Diesel uses `r2d2` connection pooling instead of SQLAlchemy's session factory
- SQLite `check_same_thread` is not needed — Diesel handles this via pool
- Diesel migrations replace Alembic — we create a single migration with all tables
- Eager loading (joinedload equivalent) done via Diesel's `belonging_to` or manual JOINs

### Scanning Thread Model
- Python uses `threading.Thread(daemon=True)` — Rust equivalent is `std::thread::spawn` or `tokio::task::spawn_blocking`
- Progress map: `Arc<Mutex<HashMap<String, ScanProgress>>>` shared between scan thread and HTTP handlers
- DB connection: scan thread gets its own connection from pool (no need for db_factory pattern)

### FFmpeg Integration
- Same subprocess approach: `std::process::Command` for ffprobe/ffmpeg
- Same JSON parsing for ffprobe output
- Same ffmpeg flags for thumbnail generation
- 30-second timeout per thumbnail via `.timeout()` or manual implementation

### Video Streaming
- Manual implementation for Range requests — actix-web doesn't have built-in range support for arbitrary files
- 8KB chunk streaming matching Python behavior
- Same content-type mapping

### Folder Browsing
- Windows: Use PowerShell with `System.Windows.Forms.FolderBrowserDialog` (avoids tkinter dependency)
- macOS: Same osascript approach
- Linux: Same zenity approach

### JSON Field Aliasing
- Python uses `Field(alias='video_metadata')` for the metadata field in VideoResponse
- Rust serde: `#[serde(rename = "metadata")]` on the field, or custom serialization
- The frontend expects `metadata` key in JSON (not `video_metadata`)

### Error Response Format
- FastAPI returns `{ "detail": "message" }` for HTTPException
- Rust must return the same shape for error responses so frontend error handling works
